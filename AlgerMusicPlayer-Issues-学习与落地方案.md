# AlgerMusicPlayer Issues 学习与落地方案

## 1. 调研范围

- 目标仓库：`algerkong/AlgerMusicPlayer`
- 调研入口：Issues 列表与近期高频问题
- 重点问题编号：
  - #605 歌曲需要手动重新解析
  - #604 / #588 / #583 播放中途停止或卡住
  - #608 音源代理 502 与可用性波动
  - #616 Safari 后台播放与切歌稳定性

## 2. 问题模式归纳

- 音源解析存在偶发失败，手动重试可成功，说明自动策略缺少“强制重解析”入口。
- 播放过程会进入 `waiting/stalled` 且无法自动恢复，缺少自愈机制。
- 单一代理或外部接口不稳定会造成链路阻塞，需要更快失败与回退。
- 用户侧需要明确可操作入口，而不是只能等待自动失败。

## 3. 本项目可迁移策略

- 增加手动“重解析”入口，支持当前歌曲强制刷新缓存后重试。
- 增加播放卡顿事件监听（`waiting`、`stalled`），触发自动重解析自愈。
- 将“解析缓存失效”显式化，提供按歌曲清理能力。
- 保持当前“固定 `m701.music.126.net` 前缀拼接”规则不变。

## 4. 落地实现清单

### 4.1 音源缓存失效能力

- 在 `src/api/music.ts` 新增：
  - `invalidateSongSourceCache(song)`：按歌曲清理 URL 缓存与候选缓存。
  - `resolvePlayableUrls(song, forceRefresh)`：支持强制绕过缓存并重新解析。

### 4.2 手动重解析能力

- 在 `src/components/MusicPlayer.vue`：
  - 增加“重解析”按钮（仅有当前歌曲时可点）。
  - 点击后调用缓存失效 + `playSong(..., { forceRefresh: true })`。

### 4.3 自动自愈能力

- 在 `audio` 元素监听：
  - `@waiting`
  - `@stalled`
- 触发后执行节流恢复：
  - 8 秒冷却窗
  - 当前在播且有用户播放意图时才恢复
  - 自动清缓存并强制重解析重播

## 5. 验证标准

- 构建通过：`npm run build`
- 基础回归：
  - 正常点播不受影响
  - 点击“重解析”可触发重新取源
  - 人为制造卡顿时可触发自动恢复逻辑

## 6. 风险与后续

- 高频 `waiting` 事件可能在弱网下被频繁触发，已用冷却窗控制。
- 后续可继续补充：
  - 恢复次数上限
  - 恢复失败后切下一曲策略
  - 恢复提示的更细粒度文案
